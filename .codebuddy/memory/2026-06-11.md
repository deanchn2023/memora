## 修复 ADP 文件上传报 InvalidRequest（16:53）

**根因**：`uploadFileToADPCOS()` 中 `IsPublic` 参数只对图片设 `true`，文件设 `false`。当 `IsPublic: false` 时，`DescribeStorageCredential` 返回的 `FileUrl` 是私有 COS 签名地址，ADP 后端无法访问 → 报 `InvalidRequest-未知错误`。

**修复**：所有文件（含图片和文档）上传到 ADP COS 时 `IsPublic` 一律设为 `true`，确保返回的 `FileUrl` 是公网可直接访问的 URL。

## 对照 Python SDK 修复 ADP 文件对话 V2 接口（17:31）

对照 ADP 官方 Python SDK 示例（`docs/pythonsdk/chat_with_file_or_img_python/main.py`）发现 5 个关键差异并修复：

1. **`DocId` → `DocBizId`**：V2 接口 `File` 对象中字段名是 `DocBizId`（Python SDK 第 332 行），不是 `DocId`
2. **`is_public` 恢复为 Python SDK 逻辑**：只有 `jpg/jpeg/png/bmp` 四种图片 `IsPublic=true`，文件 `IsPublic=false`。文件走 `docParse` 解析，不需要公网 URL
3. **`FileUrl` 自己拼接兜底**：当 `IsPublic=false` 时 `DescribeStorageCredential` 的 `FileUrl` 可能为空或私有地址，改为自己拼接 `https://{Bucket}.{Type}.{Region}.myqcloud.com{UploadPath}`
4. **`parseADPDocument` 多处修复**：
   - `session_id` 必须和后续 `chat` 请求的 `ConversationId` 保持一致（Python SDK 注释明确要求）
   - `file_name` 传完整文件名（含后缀），不是去掉后缀的名称
   - `request_id` 应等于 `session_id`（Python SDK 第 303 行）
   - SSE 响应解析修复：`payload.doc_id` + `payload.is_final`，而非 `data.doc_id` + `data.status`
5. **`uploadFileToADPCOS` 返回值增加 `type` 字段**：用于 URL 拼接

- Electron 应用中设置 `Incremental: true`，前端用 `+=` 累加是正确的
- 测试脚本未设 `Incremental` 所以返回的是完整文本，需用替换模式处理
- ADP Bot 可能返回插件授权错误（如"腾讯乐享插件需授权"），这是 Bot 配置问题非代码问题

## ADP 文件上传关键 Bug 修复（00:05）

**根因**：`parseADPDocument` 中 `cos_url` 字段用了 `cosResult.uploadPath`（仅路径，如 `/0ddf388a-xxx.docx`），但 ADP docParse API 需要完整 URL（如 `https://bucket.cos.ap-guangzhou.myqcloud.com/0ddf388a-xxx.docx`）。

**因果链**：cos_url 错误 → docParse 失败 → 无 DocId → ADP 标准模式不处理文件 → 后端收不到文档

**修复**：
1. `cos_url` 改为 `cosResult.fileUrl`（完整 URL）
2. 增加 `file_url` 字段双保险（部分 API 版本用此字段名）
3. docParse 失败时，尝试 `isPublic=true` 重新上传并再次 docParse
4. 重新上传后 URL 变为公网可访问，即使 docParse 仍失败，claw 模式也可直接通过 URL 访问文件
5. 增加 docParse 请求体和响应的详细日志
