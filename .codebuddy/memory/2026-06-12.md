## ADP V2 文件对话 4 种模式实测结果（17:45）

使用测试脚本 test/adp-file-chat-test.js 对"深圳地铁招标文件.docx"实测 4 种请求格式：

| 模式 | 请求格式 | docParse | AI 是否读到文档 | 结论 |
|------|---------|----------|---------------|------|
| A | Type:file + DocId + DocBizId, FileName 带后缀 | ✅ | ✅ 读到了，14个关键词 | **当前方式有效** |
| B | Type:file + DocId（无DocBizId）, FileName 不带后缀 | ✅ | ✅ 读到了，4个关键词 | 也有效但关键词少（可能AI回复方式不同） |
| C | Type:text + Markdown链接（Claw模式） | N/A | ❌ 超时 | Claw模式可能不适用于标准模式应用 |
| D | Type:file + DocBizId（无DocId）, FileName 带后缀 | ✅ | ❌ "请上传招标文件" | **不传DocId导致ADP读不到文档！** |

**关键发现**：
1. **DocId 字段是 ADP V2 文件对话的命脉**：模式 D 只传 DocBizId 不传 DocId，ADP 完全读不到文档 → AI 让用户"请上传"。模式 A 同时传 DocId+DocBizId 则正常。
2. **Python SDK 用 DocBizId 而不是 DocId 可能是 SDK bug**：官方 V2 文档(129202/129365)只定义了 DocId，Python SDK 用 DocBizId 是不一致的。
3. **FileName 带不带后缀都能工作**（模式 A vs B），模式 A 的效果更好（14个关键词 vs 4个）。
4. **Claw 模式的 Markdown 链接方式超时**，可能标准模式应用不支持这种方式。

**结论**：当前客户端方式（模式A）已经是正确的，之前用户遇到的"ADP当图片处理"问题，需要复测确认——可能是旧进程未重启、docParse 偶发失败、或 ADP 应用配置问题。测试脚本保留在 test/adp-file-chat-test.js，可用 `--login` 参数自动获取配置运行。

## ADP V2 文件对话官方文档 vs Python SDK 差异分析（17:03）

用户反馈：ADP 后台上传 Word 没问题，但客户端上传后 ADP 处理不正确。逐页抓取官方文档对比分析发现关键差异：

**🔴 差异 1（最可能的根因）：FileName 不应带文件后缀**
- 官方文档(107908)：`file_name: "河边的美景"` ← 明确写"不要加文件后缀"
- Python SDK：`FileName: file_name` ← Path(file_path).name 带后缀
- 我们客户端：`FileName: att.name` ← 带后缀如"xxx.docx"

**🔴 差异 2：DocBizId 字段在 V2 文档中不存在**
- HTTP SSE V2 文档(129202)：FileInfo 只有 5 个字段：FileName/FileSize/FileUrl/FileType/DocId
- WebSocket V2 文档(129365)：同上 5 个字段
- Python SDK 示例：用 DocBizId（不是 DocId）
- 我们客户端：同时传 DocId + DocBizId

**🟡 差异 3：Claw 模式文件对话用 Markdown 链接，不是 Type:file**
- Claw 模式官方示例：Contents 里用 Type:text + Text 里放 Markdown 链接 `[文件名](URL)提示词`
- 标准模式：用 Type:file + File 对象 + DocId
- 我们客户端统一用 Type:file → 如果应用是 Claw 模式，可能不被正确处理

**🟡 差异 4：V1 标准模式和 V2 接口请求体格式完全不同**
- V1：file_infos(snake_case) + content
- V2：Contents(PascalCase) + Type:file/image/text
- docParse 是 V1 接口，session_id 必须和 V2 的 ConversationId 一致（已做）

## ADP docParse 超时根因修复（14:39）

**问题**：上传 .docx 到 ADP，docParse 报 `The operation was aborted due to timeout`，拿不到 DocId，只带 FileUrl 发送 → claw 应用读不懂文件内容，AI 回"需要基于招标文件"。

**根因**：`parseADPDocument` 用 `await res.text()` 读取 docParse 的 SSE 响应。`res.text()` 会一直等到整个 HTTP 连接关闭才返回，但腾讯云 docParse 发完 `is_final` 后连接不立即关闭，导致请求一直挂起直到 120s 超时被 abort。官方 Python SDK 用 sseclient 逐事件处理、读到 is_final 立即 break 关闭连接。

**修复**：`parseADPDocument` 改用流式增量读取 `res.body.getReader()` + TextDecoder + 行缓冲，逐行解析 SSE `data:` 事件，拿到 `is_final` 立即 `done=true` 退出并 `reader.cancel()`。用 AbortController 手动控制 120s 超时（拿到结果立即 clearTimeout）。Accept 头加 `text/event-stream`。

**铁律**：ADP docParse（及所有 SSE 接口）绝不能用 res.text()/json() 一次性读，必须流式逐事件读、拿到终止事件立即 break 关闭连接，否则会等连接关闭超时。

## ADP 图片上传 Content-Type Bug 修复（00:16）

**问题**：服务器端收到的图片无法显示。

**根因**：上传图片到 COS 时 `Content-Type` 一律设为 `application/octet-stream`，导致：
1. COS 存储的 Content-Type 是 `application/octet-stream`（而非 `image/png` 等）
2. ADP 服务端根据 Content-Type 判断文件类型，octet-stream 的图片不被识别为图片
3. 图片 URL 虽然正确，但 ADP 不认为是图片，无法显示

**修复**（4 处）：
1. `uploadFileToADPCOS` 方式 A（UploadUrl）：`Content-Type` 从 `application/octet-stream` 改为根据文件扩展名映射正确的 MIME Type
2. `uploadFileToADPCOS` 方式 B（COS SDK）：`putObject` 增加 `ContentType` 参数
3. `isPublic=true` 重新上传 UploadUrl 方式：同样用正确的 MIME Type
4. `isPublic=true` 重新上传 COS SDK 方式：`putObject` 增加 `ContentType` 参数

新增 MIME Type 映射表：jpg→image/jpeg, png→image/png, gif→image/gif, bmp→image/bmp, webp→image/webp, heic→image/heic, pdf→application/pdf, docx→application/vnd.openxmlformats-officedocument.wordprocessingml.document 等

## 记事本图片同步服务器端 400 错误根因修复（00:41）

**问题**：记事本图片同步到服务器后无法显示。服务器端 `image_path` 存储了本地格式路径（如 `images/4bdfb3f25f27a1a3_3024x1766.png`）。

1. 服务器部署代码 (`/root/memora-modular/routes/sync.js`) 的图片上传路由使用了 `requireActiveDevice` 中间件

4. 上传失败后，笔记元数据仍被推送到服务器，`image_path` 使用了本地格式路径
5. 服务器端无法解析本地格式路径，导致图片 404

**修复**（3 处）：
1. **客户端 `_uploadNoteImageViaHttp`**：添加 `form.append('device_id', getDeviceFingerprint())`
2. **客户端 `sync:push` 处理**：图片上传失败时将 `image_path` 设为空字符串（而非保留本地路径）
3. **客户端 `saveClipboardImage` 触发推送**：`imagePath` 只用 `serverImagePath || ''`，不再用 `localImagePath` 兜底
4. **本地 `config-server/sync-routes.js`**：添加 `requireActiveDevice` 中间件和函数定义，与部署代码同步

**关键经验**：本地 workspace 的 `config-server/` 代码与服务器实际部署的 `/root/memora-modular/` 代码可能不同步，排查服务器端问题必须检查部署代码。

## ADP Claw 模式文件传递方式根本性修复（22:15）

**根因发现**：查阅官方文档 https://cloud.tencent.com/document/product/1759/107908 后确认，Claw 模式和标准模式的文件传递方式完全不同：
- 标准模式：Type:file + DocId（需 docParse 解析文档获取 DocId）
- **Claw 模式：Markdown 链接嵌入 Type:text（不需要 docParse！）**
  - 文档：`[文件名](COS预签名URL)` 嵌入 Text
  - 图片：`![](图片URL)` 嵌入 Text

之前一直用标准模式的方式（Type:file + DocId）给 Claw 模式传文件，所以 ADP 读不到内容。MD/TXT 正常是因为直接注入文本，不走 COS+docParse。

**修复内容**：
1. COS 上传成功后，跳过 docParse，直接用 Markdown 链接方式
2. 图片也改为 `![](url)` Markdown 格式嵌入 Type:text
3. File Share 降级路径同步改为 Markdown 方式
4. 移除 10 秒等待（docParse 不再调用）
5. 更新诊断日志适配 Claw 模式

**ADP Widget SDK**：V2 接口 Content 支持 Widget 类型（WidgetId/WidgetRunId/State/View），官方提供了 ADP-Widget SDK（Web Components 标准）用于渲染交互组件。Widget 交互通过 Type:widget_action 回传。
