## 剪贴板图片同步 Bug 修复

### 问题
剪贴板图片保存后，笔记元数据（含 image_path）推送到了服务端，但实际图片文件没有上传。服务端的 `image_path` 字段仍是本地路径格式 `images/xxx.png`，而非服务端路径格式 `{userId}/xxx.png`。

### 根因分析
1. `saveClipboardImage()` 只保存本地文件和创建笔记，不触发图片上传
2. 图片上传依赖 fullSync（每 1 分钟自动触发），但 fullSync 的 `uploadNoteImage()` 可能因 Node.js 原生 fetch 与 form-data 流兼容性问题失败
3. 上传失败时，笔记仍以本地 image_path 推送到服务端，导致服务端存储了无效路径
4. 无即时同步触发，剪贴板图片保存后无法第一时间同步

### 修复内容
1. **`saveClipboardImage` 改为 async**：保存本地文件后立即尝试上传图片到服务端
   - 上传成功 → 用 server_path 创建笔记
   - 上传失败 → 仍用本地路径创建笔记（下次 fullSync 重试）
   - 未登录 → 记录日志，等待下次 fullSync

2. **`uploadNoteImage` 增加 HTTP 降级方案**：
   - 先尝试原生 fetch + form-data 流式上传
   - 失败后降级到 `http/https` 模块 + form.pipe(req)
   - 增加 30 秒超时和详细错误日志

3. **fullSync 图片上传失败时阻止推送**：
   - 图片文件丢失或上传失败 → 从 data.changes.notes 中移除该笔记
   - 避免将本地 image_path 推送到服务端

4. **增加即时同步触发**：
   - 新增 `sync:trigger-push` IPC 事件
   - preload.js 增加 `onSyncTriggerPush` 监听
   - sync.js init() 中监听该事件并调用 `markDirty`

5. **增强日志**：fullSync 图片上传步骤增加文件大小、路径等详细信息

### 涉及文件
- `main.js`: saveClipboardImage、uploadNoteImage、_uploadNoteImageViaHttp、fullSync 图片上传段
- `preload.js`: 新增 onSyncTriggerPush
- `src/scripts/sync.js`: init() 中监听 sync:trigger-push
